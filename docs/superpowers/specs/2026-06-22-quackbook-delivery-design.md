# quackbook — Дизайн деливери v1

- **Дата:** 2026-06-22
- **Статус:** утверждён к реализации (брейншторминг завершён, переход к плану)
- **Источник скоупа:** `scope-quackbook-v1.md` (единственный источник истины по продукту)
- **Правила:** `CLAUDE.md`

## Контекст и цель

quackbook — браузерный аналитический ноутбук на DuckDB-WASM. Два слоя: **данные** (drop CSV/Parquet → схема в рейле → SQL с join/union → Профиль) и **ноутбук** (закрепляешь результат виджетом → пишешь текст → собираешь вертикальный нарративный отчёт). Всё на клиенте, без бэкенда, статика на GitHub Pages. v1 — только локально загруженные файлы.

Этот документ — **план деливери**: решения по каркасу и стеку, границы модулей и нарезка v1 на вехи с критериями готовности. Реализационный план (по шагам) пишется отдельно после утверждения этого спека.

## Принятые решения

1. **Каркас: React + TypeScript + Vite.** Ноутбук — список блоков с пер-блочным стейтом, табы с независимым стейтом редактора/результата, виджет-карточки — это «заметный компонентный стейт» (скоуп, строка 98). Компонентная модель снимает ручной DOM-плумбинг. Бандл-оверхед React (~45KB gz) ничтожен рядом с DuckDB-WASM (мегабайты), холодный старт упирается в инстанциацию WASM. Edge проекта — движок данных, а не layout-плумбинг; React освобождает усилия под edge. TS усиливает TDD на логике (билдеры SQL, инференс, `TRY_CAST`, сериализация).

2. **Навигация: гибрид.** Верхний тогл Исследование/Отчёт + табы редакторов под разные датасеты внутри режима исследования. Решает расхождение скоуп↔мокап (скоуп описывает Sublime-табы, мокап — mode-toggle): берём оба уровня.

3. **Key-хинт — вырезан полностью** (ни v1, ни v1.5). Авто-подсказка «events.user_id ↔ users.id, сджойнить?» зависит от готового Профиля + кросс-табличного матчинга кардинальностей — слишком дорого относительно ценности. **Это переопределяет скоуп** (строка 67 относила хинт к v1). Ручной `JOIN`/`UNION` между CSV через SQL остаётся в ядре v1 — он нативен для общей in-memory DuckDB и от хинта не зависит.

4. **Темп: качество важнее сроков.** Вехи делаем полными и полированными; календарных дат не ставим.

5. **Подход к нарезке: сквозной скелет, затем наращивание (depth-first).** Тонкая сквозная нить через все слои (M0), затем по одной полной способности. Снимает технический риск движка первым; каждая веха демонстрируема и задеплоена.

6. **Single-thread DuckDB-WASM.** Целимся в однопоточную сборку (мокап, статус-бар: «single-thread»). Ей не нужны COOP/COEP-заголовки (cross-origin isolation), которые GitHub Pages не умеет выставлять. Это подтверждает совместимость движка со статикой на Pages.

### Рекомендуемый стек (за границами модулей, заменяемо)

- **Чарты:** Observable Plot (встроенные bar/line/histogram, рендер в SVG, хорошо с колоночными данными). Прячем за компонентом `<Chart>` — смена на Recharts остаётся дешёвой, не one-way door.
- **Грид результата:** `@tanstack/react-virtual` поверх простой таблицы (read-only, плотный).
- **SQL-редактор:** CodeMirror 6 + SQL-режим (автокомплит по именам загруженных таблиц/колонок — nice-to-have).
- **Markdown:** `marked` (как в скоупе).
- **Перестановка блоков:** `dnd-kit` (вертикальный sortable).
- **Стор сессии:** Zustand (мал, тестируется вне React). Альтернатива — context + reducer.
- **Экспорт:** print-CSS + `window.print` (дёшево); `html2pdf` только если упрёмся.

## Расхождения со скоупом/мокапом (зафиксированы)

- **Key-хинт:** скоуп (строка 67) относил к v1 — мы **вырезаем совсем** (решение 3).
- **Навигация:** скоуп — Sublime-табы; мокап — mode-toggle. Берём **гибрид** (решение 2).
- **Имя:** мокап — «quackpad»; скоуп/README — «quackbook». Источник истины — **quackbook**.

## Архитектура — границы модулей

Четыре зоны. Каждая имеет одну цель, чёткий интерфейс, тестируется отдельно.

### `db/` — единственный, кто говорит с DuckDB
DuckDB-WASM (single-thread) в Web Worker через `AsyncDuckDB`. Тонкий клиент на главном потоке: `init()`, `registerFile(name, File)`, `query(sql) → Arrow.Table`, `exec(ddl)`. Всё остальное ходит только через него. Воркер почти не пишем руками — используем готовый воркер-скрипт duckdb-wasm.

### `core/` — чистая логика, сердце TDD
Без DOM и живого DuckDB, только вход→выход:
- **SQL-билдеры:** `buildSelectAll(table)`; билдеры запросов профиля (`SUMMARIZE`, top-values `GROUP BY`); **билдер DDL материализации** из конфига схемы (`CREATE TABLE ... AS SELECT TRY_CAST(col AS type) ...`).
- **Инференс схемы:** парсинг вывода `sniff_csv` → дескрипторы колонок; baseline `all_varchar`; предложение типов one-click.
- **`TRY_CAST`-логика + счётчик «N → NULL»:** построение каст-выражения и запроса подсчёта, интерпретация результата.
- **Шейпинг профиля:** Arrow-результат `SUMMARIZE` + топ-значения → вью-модели карточек (бары для категориальных, гистограмма для числовых).
- **Pruning-детект:** по тексту SQL + известной схеме определить, какие колонки таблицы читает запрос (подсветка в рейле). Лёгкий парс идентификаторов по именам колонок, не полный SQL-парсер.
- **Сериализация отчёта:** документ-блоки (текст; виджет `{title, sql, datasetNames, vizType, caption}`) ↔ версионированный JSON. Save/load **структуры**, без данных.

### `state/` — сессионный стор (Zustand)
Загруженные датасеты (имя таблицы, метаданные файла, конфиг схемы), табы исследования (id, датасет, sql, ссылка на последний результат, активный вид), документ отчёта (упорядоченные блоки), активный режим/таб. **Только в памяти** (сессия жива, пока вкладка открыта); Reset чистит; reload теряет таблицы (честная оговорка скоупа).

### `features/` + `components/` — React UI
- **Shell:** топбар, тогл режимов, Reset, контекст сессии, пилюля `local`, сохранить.
- **Rail** (общий для режимов): список источников, схема с pruning-подсветкой, кнопка профиля.
- **Explore:** менеджер табов (редактор под датасет), CodeMirror, run, панель результата (таблица/график/профиль), pin-в-отчёт.
- **Report:** список блоков, текст-блок (`marked`), виджет-блок (SQL+датасет+виз+подпись), вертикальная перестановка, add/export.
- **Общий рендер результата:** виртуализированный грид, чарты (`<Chart>`), карточки профиля.

## Поток данных

drop файла → `registerFile` → (CSV) `sniff_csv` → схема в рейле (правится) → материализация с `TRY_CAST` + счётчик null → клик по датасету = новый таб `SELECT * FROM t` → run → Arrow → грид/чарт + pruning-подсветка → профиль по кнопке → pin результата = виджет-блок → режим Отчёт перезапускает SQL виджетов по таблицам в памяти → save (JSON структуры) / export (HTML/PDF) → reload: таблиц нет, при открытии отчёта просим подгрузить нужные файлы (виджет знает имя датасета), затем перезапуск.

## Обработка ошибок (минимум, по CLAUDE.md rule 2)

Только реальные сценарии:
- Ошибка SQL из DuckDB → показать текст в панели результата, не падать.
- Сбой `sniff_csv` → сообщение + остаёмся на `all_varchar` baseline.
- Холодный старт WASM → честный лоадер.
- Reload с отсутствующими датасетами для отчёта → подсказка, какой файл нужен (виджет знает имя датасета).

`TRY_CAST`→NULL — не ошибка, а фича-счётчик. Никакой спекулятивной обработки под несуществующие сценарии.

## Стратегия тестирования (Vitest, TDD)

Red-green-refactor для всего `core/`: билдеры SQL, инференс, `TRY_CAST`-логика, шейпинг профиля, pruning-детект, сериализация отчёта. `db/` — пара интеграционных smoke-тестов против реального DuckDB-WASM в Node. Презентацию (CSS/раскладка/цвета) тестами не покрываем — глазами (честная граница CLAUDE.md). Точечно — RTL-тесты на стейтовое поведение (переключение табов, pin добавляет блок).

## Дорожная карта вех

Каждая веха: зелёные тесты, задеплоена на Pages, самостоятельно демонстрируема. Порядок строгий M0→M5. Вес — относительный (без календаря).

### M0 — Спина (walking skeleton) · вес: M
Доказать движок насквозь и получить задеплоенную оболочку.
- Каркас Vite+React+TS, ESLint/Prettier, Vitest.
- DuckDB-WASM (single-thread) в Web Worker; клиент `db/`: `init / registerFile / query`.
- Drop одного CSV → регистрация как `all_varchar` → фикс. `SELECT * FROM t LIMIT n` → сырая таблица.
- Честный лоадер на инициализации WASM.
- Деплой на GH Pages через GitHub Actions (правильный `base` в Vite).

**Готово, когда:** на задеплоенном URL бросаю CSV — вижу строки; виден лоадер; reload даёт пустое состояние.
**TDD:** `buildSelectAll(table)`; smoke-интеграция `db/` в Node (мини-CSV → query → проверка строк).
**Снимает риск:** WASM-в-воркере + Arrow + Vite + статика Pages + single-thread (без COOP/COEP).

### M1 — Режим исследования · вес: L (самая большая)
Настоящий цикл исследования над загруженными файлами.
- Shell: топбар, тогл Исследование/Отчёт (Отчёт пока заглушка), пилюля `local`, **Reset** (drop таблиц, очистка сессии).
- Rail: список источников (мульти-файл CSV+Parquet), схема из `information_schema` (имя+тип).
- Explore: **менеджер табов** — клик по датасету открывает/фокусирует таб с `SELECT * FROM <table>`; CodeMirror 6 на таб; Run (⌘↵).
- Панель результата: виртуализированный грид + мета (строки, мс); тоглы Таблица/График; **базовый чарт** (bar + line).
- **Мульти-файл JOIN/UNION** через SQL (нативно, общая БД).
- **Pruning-подсветка**: рейл подсвечивает колонки, что читает текущий запрос.
- Сессионный стор (Zustand): датасеты, табы, активный таб/вид.

**Готово, когда:** гружу 2 файла, открываю табы, пишу запрос с `JOIN` между ними, вижу результат в гриде и на чарте, рейл подсвечивает used-колонки, Reset чистит всё.
**TDD:** pruning-детект (SQL+схема → used-колонки); операции стора табов; шейпинг Arrow→строки грида.

### M2 — Схема и типы · вес: L
Превратить сырой `all_varchar` в типизированные таблицы безопасно.
- `sniff_csv`-инференс при загрузке → предложенные типы; one-click «типы».
- Редактор схемы: rename, тип, формат даты/decimal, nullstr, include/exclude по колонке.
- Материализация: `CREATE TABLE ... AS SELECT TRY_CAST(...)` из конфига схемы; ре-материализация на apply.
- `TRY_CAST` + счётчик «N → NULL» на колонку (маркер ⚠ из мокапа).

**Готово, когда:** гружу «грязный» CSV, вижу инференс, меняю тип/формат колонки, applyю, вижу счётчик N→NULL по неудачным кастам, запрашиваю типизированную таблицу.
**TDD:** маппинг инференса (`sniff_csv`→дескрипторы); билдер DDL с `TRY_CAST` + include/exclude + rename; билдер запроса подсчёта N→NULL + интерпретация; логика предложения типов.

### M3 — Профиль · вес: M
Видеть, что внутри колонок, одним проходом.
- Прогон профиля: `SUMMARIZE` (min/median/max/null%/uniq) + точечные `GROUP BY` для топ-значений категориальных.
- Вид профиля: карточки — бары для категориальных, гистограмма для числовых; вкладка «профиль» в панели + кнопка «профиль источника» в рейле.
- Чарт: добавляется histogram.

**Готово, когда:** жму профиль → карточки с distinct-каунтами, бары топ-значений, гистограммы числовых + min/median/max, маркеры null.
**TDD:** билдеры запросов профиля (`SUMMARIZE` + топ-значения); шейпинг профиля (Arrow→вью-модели карточек, ветка категориальный/числовой, топ-N + «+N ещё»).

### M4 — Ноутбук (Отчёт) · вес: L · ← здесь появляется полный демо-нарратив
Портфолио-payoff — нарративный отчёт.
- Режим Отчёт реальный: ноутбук = вертикальный стек блоков.
- Текст-блок (markdown через `marked`, редактируемый).
- Виджет-блок: pin результата из Explore → `{title, sql, datasetNames, vizType, caption}`; сворачиваемый SQL + имя датасета; тогл Таблица/График; подпись; перезапуск SQL по таблицам в памяти при открытии отчёта.
- Вертикальная перестановка (`dnd-kit`), кнопки + текст / + запрос.
- Подсветка рейла следует активному блоку отчёта.
- Save/load **структуры** отчёта в JSON + реГидрация: при load просим подгрузить нужные датасеты (виджет знает имя), затем перезапуск.

**Готово, когда:** закрепляю 2–3 результата, пишу текст между ними, переставляю, виджеты пересчитываются; сохраняю JSON, перезагружаю страницу, загружаю JSON → просит подгрузить датасеты → виджеты пересчитались.
**TDD:** round-trip сериализации отчёта (doc↔JSON, версионируется); логика «какие датасеты нужны виджету»; операции порядка блоков.

### M5 — Экспорт и полировка · вес: M
Отчуждаемый артефакт + полировка.
- Экспорт HTML (self-contained, инлайн-стили, отрендеренный отчёт).
- Экспорт PDF (print-CSS + `window.print`).
- Полировка: лоадер, пустые состояния, поверхности ошибок, ⌘↵, фокус/доступность, reduced-motion, адаптив.
- README «Старт» + команды (`dev/test/build/deploy`).

**Готово, когда:** экспортирую отчёт в standalone-HTML, открывается офлайн и выглядит верно; печатаю в PDF; полировка проверена глазами.
**TDD:** сериализация HTML-экспорта (doc→self-contained HTML-строка). PDF — print-CSS, глазами.

**Линия MVP-демо:** конец **M4** (полный нарратив работает). **M5** доводит до отчуждаемого и полированного.

## Вне скоупа v1 (firewall — не строить без явного запроса)

- Dashboard-грид / freeform-канвас / drag-resize плитки / многоколоночная раскладка. Отчёт — вертикальный стек, ширина одна.
- Визуальный join-builder (перетащи колонку на колонку).
- Key-хинт (вырезан совсем — см. решение 3).
- Персист данных (OPFS), правка значений per-cell, деривативные колонки.
- Мультипользовательность, шеринг с правами, коллаборация.
- Таб «План» (EXPLAIN ANALYZE).

## Открытые вопросы / риски

- **Данные испаряются при reload** — v1: «сохранить отчёт» сохраняет структуру, при открытии просим подгрузить датасеты заново (M4). OPFS-персист — v1.5.
- **Layout creep** — как только блоки двигаются, тянет в грид. Firewall «вертикаль, не грид» держим жёстко.
- **Bundle size / холодный старт** — честный лоадер (M0), single-thread сборка.
- **Чарт-либа** — Observable Plot за границей `<Chart>`; если упрёмся в фит — смена на Recharts дешёвая.
